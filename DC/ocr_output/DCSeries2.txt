1. Illustrate Richart Algorithm for achieving mutual exclusion
Think of mutual exclusion like waiting in line to use an ATM. Only one person can use it ata
time. Similarly, in a distributed system, multiple computers (processes) may need to access a
shared resource (like a file or a database), but only one should access it at a time.
The Ricart-Agrawala Algorithm helps computers take turns fairly and efficiently. Here's how
it works:
How It Works (Step-by-Step)
@ Step 1: Asking for Permission
e Ifa process (computer) wants to use a resource, it sends a "REQUEST" message to all
other processes.
e This message has a timestamp (like writing down the time you arrived in line).
) Step 2: Receiving a Request
e When another process receives a REQUEST, it checks:
e "Am | using the resource?"
e If No — It sends back "REPLY" (Yes, you can use it) immediately.
e If Yes — It waits and sends REPLY later after it's done.
e If it's also waiting for the resource, it compares timestamps:
e Smaller timestamp (arrived first) = That process gets priority.
e Larger timestamp (arrived later) — It waits.

===== PAGE 5 =====
Step 3: Using the Resource
e The process enters the critical section (CS) (uses the resource) only after receiving
REPLYs from all other processes.
e This ensures only one process uses it at a time.
@ Step 4: Releasing the Resource
e Once it's done, it sends REPLYs to any waiting processes.
e The next process in line can now enter the CS.

3. Explain checkpointing and rollback recovery in detail
Imagine you are playing a video game, and you reach an important level. To avoid losing
progress if something goes wrong, you save the game at certain points. If you fail later, you
can reload the saved checkpoint instead of starting from scratch.
This is exactly how Checkpointing and Rollback Recovery work in distributed systems to
handle failures!
What is Checkpointing?
Checkpointing is the process of saving the state of a process at a certain point so that ifa
failure occurs, it can restart from that point instead of starting over.
Why Use Checkpoints?
e Reduces the amount of lost work after a failure.
e Helps systems recover quickly instead of re-executing everything.
e Saves CPU time and network resources.

===== PAGE 7 =====
XX Example:
A bank transaction system saves its progress after every 100 transactions. If the system
crashes after 150 transactions, it restores the last saved state (100 transactions) and
reprocesses the last 50 instead of all 150.
What is Rollback Recovery?
Rollback Recovery is the process of restoring a system to a previous consistent
checkpoint after a failure.
Steps in Rollback Recovery:
1. Detect Failure
e The system notices that a process has failed.
2. Restore Checkpoint {J
e The system reloads the last saved checkpoint for that process.
3. Re-execute
e The process continues from where it left off, avoiding major data loss.
< Example:
If an airline booking system crashes after booking 50 tickets, it restores the last saved
checkpoint (e.g., after booking 40 tickets) and reprocesses the last 10 tickets.
Types of Checkpointing (How to Save Progress)
There are three main ways to implement checkpointing:
A. Uncoordinated Checkpointing (Independent Checkpoints)
e Each process saves its own state independently.
e Problem: It may lead to a Domino Effect (rolling back too far).
< Example:
If process P1 rolls back, but it has sent data to P2, then P2 must also roll back, and so on.
This can cause the system to restart from a very old state.
B. Coordinated Checkpointing (Planned Checkpoints)
e All processes coordinate and save a consistent global state.
e Prevents the Domino Effect and ensures all processes are in sync.

e Uses a coordinator process to signal all processes to take checkpoints together.
“SX Example:
An online shopping system saves a checkpoint every hour across all servers. If one server
crashes, it restores all servers to the last saved state to maintain consistency.
C. Communication-Induced Checkpointing (Smart Checkpointing)

e Checkpoints are automatically triggered when needed.

e Prevents unnecessary rollbacks and reduces overhead.

e Uses extra information in messages to decide when to save a checkpoint.

“SX Example:

A cloud storage service automatically saves checkpoints when many files are being
uploaded, ensuring smooth recovery if a failure occurs.

What is a Consensus Algorithm?
A consensus algorithm helps multiple processes (computers) agree on a single value ina
distributed system. This is important when some processes fail or crash but the system must
continue working.
Understanding Synchronous Systems
A synchronous system means:
v Time is predictable — All messages are delivered within a known time.
v Processing is predictable — Each process takes a known time to compute.
v Failures can be detected — If a process doesn’t respond in time, we assume it has crashed.
¢ Example: Imagine a group of friends voting on where to eat. Everyone must answer within
10 seconds, or they are considered unavailable.

===== PAGE 10 =====
Consensus in the Presence of Crash Failures
In a synchronous system, crash failures happen when some processes stop working but
don’t send incorrect data. The goal of the consensus algorithm is to let the remaining
processes agree on a decision, even if some fail.
Steps to Reach Consensus in Crash Failures:

1. Each process proposes a value (e.g., "Let's eat pizza" or "Let's eat burgers").

2. Processes exchange values with each other within a fixed time.

3. If a process crashes, it stops responding, but the remaining processes continue.

4. Majority rule: If more than half of the processes agree on a value, that value is chosen.

5. Final Decision: All non-crashed processes adopt the agreed-upon value.

Whole-File Caching for Faster Access
e When a user opens a file, the entire file is downloaded to their computer.

e Even if the computer restarts, cached files remain available.
Whole-File Serving
e AFS sends the full file instead of small parts at a time.
e This reduces network traffic and makes file access more efficient.

¢ Scenario: A User Opens a File

3. The file is stored (cached) locally on the user's computer.
4. The user can read, write, or edit the file from the local copy.
5. When the user closes the file, any changes are sent back to the server.
6. The AFS server updates the main file so others can see the changes.

8. Explain the google file system
The Google File System (GFS) is a distributed file system developed by Google to handle
large-scale data processing across thousands of machines. It is designed to store huge files
efficiently and support applications like Google Search, YouTube, and Google Drive.
GFS follows a Master-Slave Architecture with three main components:
Master Server
e Acts as the brain of GFS.
e Manages metadata (file locations, chunk details).
e Keeps track of which Chunk Server has which file pieces.
e Does not store actual file data.
Chunk Servers
e Store large file chunks (typically 64MB each).
e Multiple copies (replicas) of each chunk are stored for fault tolerance.
e Handles read and write requests from clients.
Clients
e Users or applications that request files from GFS.
e First, they ask the Master Server for chunk locations.
e Then, they directly retrieve the file chunks from Chunk Servers.
How GFS Reads a File __|

===== PAGE 15 =====
¢ Example: A user wants to read a file stored in GFS.
1. Client asks the Master Server for file chunk locations.
2. Master Server replies with the list of Chunk Servers that store the chunks.
3. Client directly fetches the file from the Chunk Servers.
How GFS Writes a File #
¢ Example: A user wants to save a file in GFS.
1. Client sends a write request to the Master Server.
2. The Master tells the client which Chunk Server to write to.
3. The file is broken into chunks (64MB each) and saved on multiple Chunk Servers.
4. Replication happens automatically to ensure data is not lost.

11. Explain different file system requirements
A distributed file system (DFS) allows users to access files from multiple computers as if they
were Stored locally. To ensure efficiency and usability, the system must meet certain
requirements.
1. Access Transparency
e Users and programs should not need to know whether a file is stored locally or remotely.
e The same file operations should work for both.
e Example: Accessing a file on Google Drive should feel the same as opening a file stored on
a personal computer.
2. Location Transparency
e Files can be moved between servers, but their path remains unchanged.
e Users do not need to know where the file is physically stored.
e Example: A video on a streaming platform may move to different data centers, but users
can still access it using the same link.
3. Mobility Transparency
e Files can be moved without requiring changes in client applications or system settings.
e Ensures that files remain accessible even when they are relocated.
e Example: A company might move employee files from one server to another without
employees noticing any change.
4. Performance Transparency
e The system should maintain stable performance even when the load on servers varies.
e Example: A cloud storage service should provide smooth access to files even when many
users are active.
5. Scaling Transparency
e The system should be able to expand and handle an increasing number of users and data
without major changes.

===== PAGE 19 =====
e Example: A cloud-based file service should function efficiently whether it serves ten users
or a million users.
6. Concurrent File Updates
e Multiple users should be able to update files simultaneously without conflicts.
e Example: In a shared document, two users editing at the same time should not overwrite
each other’s changes.
7. File Replication
e The system should maintain multiple copies of files across different locations.
e Helps in load balancing and fault tolerance.
e Example: A distributed file system storing data on multiple servers ensures availability even
if one server fails.
8. Hardware and Operating System Heterogeneity
e The system should work across different operating systems and hardware.
e Example: A file service should be accessible from Windows, Linux, and macOS without
compatibility issues.
9. Fault Tolerance
e The system should continue functioning even if some servers or clients fail.
e Example: If one storage server crashes, another should take over automatically to prevent
data loss.
10. Consistency
e When files are updated, all copies should reflect the latest changes.
e There might be delays in propagating updates across different sites.
e Example: When an email is deleted from one device, it should also disappear from all other
devices.
11. Security
e The system should protect data using authentication, access control, and encryption.

===== PAGE 20 =====
e Example: Only authorized users should be able to access confidential company files, and
data should be encrypted to prevent unauthorized access.

12. Efficiency

e The system should provide high performance comparable to traditional file systems.

e Example: Opening and saving files in a distributed system should be as fast as working

with local files.

12. Explain different models of deadlock
Deadlock occurs when processes wait indefinitely for resources that are held by other
processes. Distributed systems allow different ways to request resources, leading to different
models of deadlock.
1. Single-Resource Model

e Aprocess can request only one resource at a time.

e The system grants the resource only if it is available.

e Ifa process is already holding a resource, it must release it before requesting another.
Deadlock Detection:

e Ina Wait-For Graph (WFG), each node can have at most one outgoing edge.

e If acycle is present, a deadlock has occurred.
Example:

e Process A requests Resource 1.

e Process B requests Resource 2.

e If Ais waiting for B and B is waiting for A, a cycle forms — Deadlock!
2. AND Model

e Aprocess can request multiple resources at the same time.

e The request is granted only if all requested resources are available.

e The requested resources can be on different locations (Servers).

===== PAGE 21 =====
Deadlock Detection:
e Ina WEG, each node can have multiple outgoing edges.
e If acycle exists, a deadlock has occurred.
Example:
e Aprocess requests Printer AND Disk Space.
e If one is available but the other is not, the process waits indefinitely.
3. OR Model
e Aprocess requests multiple resources, but it only needs any one of them to proceed.
e If at least one resource is granted, the process continues execution.
Deadlock Detection:
e Acycle in the WFG does not always mean a deadlock because a process may still
proceed if one of the requested resources is available.
Example:
e Aprocess requests "Printer OR Scanner".
e If at least one is available, the request is satisfied.
e If neither is available, the process waits until one is free.
4. AND-OR Model
e Acombination of the AND model and OR model.
e Aprocess can request some resources together (AND) while having options for others
(OR).
Example:
e Aprocess requests "Disk AND (Printer OR Scanner)".
e It must get Disk, but it can proceed with either a Printer or a Scanner.
e Deadlock depends on the specific resource combinations and availability.
5, P-out-of-Q Model (Subset of AND-OR Model)

===== PAGE 22 =====
e Aprocess requests any P resources out of Q available resources.
e If at least P resources are available, the request is granted.
Example:
e Acloud server needs any 2 out of 5 available CPU cores to process a request.
e If atleast 2 cores are available, execution proceeds.
e If fewer than 2 cores are available, the process waits.
13. Explain different types of messages
In a distributed system, messages are used for communication between processes. However,
when failures or rollbacks occur, different types of message inconsistencies can arise.
1. In-Transit Messages
e Messages that have been sent but not yet received by the destination process.
e These messages do not cause inconsistency because they will eventually be delivered.
e Example:
e Process A sends a message to Process B, but before B receives it, A crashes.
e When A recovers, the message is still in transit and will be delivered later.
2. Lost Messages
e Messages that were sent but were never received due to a rollback.
e The sender does not roll back, but the receiver rolls back to a state before the message
was received.
e Example:
e Process A sends a message to Process B.
e Process B rolls back to an earlier state before it received the message.
e The message is lost because B no longer remembers receiving it.
3. Delayed Messages
e Messages that were sent, but their reception was not recorded because:
e The receiver was down when the message arrived.

===== PAGE 23 =====
e The receiver rolled back before processing the message.
e Example:
e Process A sends a message to Process B.
e Before B receives it, B crashes and rolls back.
e The message is now delayed and might be received later or lost.
4, Orphan Messages
e Messages where the receive event is recorded, but the send event is not recorded.
e This happens when a rollback undoes the send event, but the receive event remains.
Example:
e Process A sends a message to Process B.
e Later, Arolls back to a state before the message was sent.
e Now, B thinks it received a message that was never sent.
5. Duplicate Messages
e Messages that are resent due to rollback and replay mechanisms.
e This happens when a message is logged, and after recovery, it is resent even though the
receiver already received it.
e Example:
e Process A sends a message to Process B.
e Both processes roll back to earlier states before the message was sent and
received.
e When A resends the message, B might receive it twice, creating a duplicate
message.
14. Consistent and inconsistent states
1. Consistent State
A consistent state is a state where:

===== PAGE 24 =====
e Ifa process has received a message, then the corresponding send event must have
happened.
e The system is in a state that could have naturally occurred during failure-free
execution
Example of a Consistent State
e Process A sends message m1 to Process B.
e Process B receives m1 before a failure occurs.
e After recovery, the system still shows that A sent m1 and B received it.
Since both the send and receive events are recorded, this state is consistent.
2. Inconsistent State
An inconsistent state is a state where:
e Aprocess shows that it received a message, but the corresponding send event is
missing.
e This situation is impossible in a correct failure-free execution.
Example of an Inconsistent State
e Process A sends message m2 to Process B.
e Process B records that it received m2, but Process A rolls back to a checkpoint before
sending m2.
e Now, the system shows B received m2, but A never sent it.
Since this state cannot occur in normal execution, it is inconsistent.
